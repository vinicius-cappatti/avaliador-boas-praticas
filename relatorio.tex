\begin{abstract}
This paper presents an artificial intelligence agent for Python code review
based on programming best practices. The solution combines deterministic static
analysis with pycodestyle and natural-language feedback generation
through a language model accessed by Ollama. The implemented pipeline
supports local files, local directories, and public GitHub repositories. In a
25-file evaluation dataset, the deterministic stage found 217 PEP 8 violations
in 12 files, while 13 files had no violations. Results indicate that grounding
the model response in static-analysis evidence improves traceability and makes
the generated feedback more useful for programming education.
\end{abstract}

\begin{resumo}
Este artigo apresenta um agente de inteligência artificial para revisão de
código Python baseado em boas práticas de programação. A solução combina análise
estática determinística com pycodestyle e geração de feedback em
linguagem natural por meio de um modelo de linguagem acessado via
Ollama. O pipeline implementado aceita arquivos locais, diretórios
locais e repositórios públicos do GitHub. Em uma base de avaliação com 25
arquivos, a etapa determinística identificou 217 violações PEP 8 em 12 arquivos,
enquanto 13 arquivos não apresentaram violações. Os resultados indicam que
ancorar a resposta do modelo em evidências de análise estática aumenta a
rastreabilidade e torna o feedback mais útil em contextos de aprendizagem de
programação.
\end{resumo}

\section{Introdução}

A revisão de código é uma prática recorrente no desenvolvimento de software e
tem relação direta com legibilidade, manutenibilidade, padronização e redução
de defeitos. Em equipes profissionais, revisões ajudam a compartilhar
conhecimento e a detectar problemas antes da integração ao projeto. Em ambientes
educacionais, entretanto, a revisão manual contínua costuma ser limitada pelo
número de estudantes, pelo tempo disponível dos professores e pela dificuldade
de oferecer retorno individualizado em ciclos curtos. Como consequência,
estudantes iniciantes podem manter vícios de formatação, nomenclatura e
organização mesmo quando seus programas apresentam comportamento funcionalmente
correto.

No caso da linguagem Python, a PEP 8 define convenções amplamente adotadas para
organização visual, espaçamento, nomenclatura e estilo de código
\cite{vanrossum2001pep8}. Tais convenções não garantem, isoladamente, a correção
funcional de um programa, mas contribuem para que o código seja mais fácil de
ler, revisar e manter. Ferramentas de análise estática, como
pycodestyle, conseguem verificar parte dessas regras de forma
determinística e reproduzível \cite{pycqa2026pycodestyle}. Por outro lado, a
saída dessas ferramentas é frequentemente técnica e sintética, o que pode
dificultar seu uso por estudantes que ainda não dominam a terminologia das
regras PEP 8.

O problema de pesquisa abordado neste trabalho é, portanto, como transformar
violações objetivas de estilo em feedback compreensível, rastreável e útil para
aprendizes de programação. A contribuição proposta consiste em um agente de IA
que combina a precisão de um validador estático com a capacidade explicativa de
um modelo de linguagem. Nessa arquitetura, o pycodestyle atua como
fonte determinística das violações, enquanto o modelo de linguagem organiza as
informações em um relatório didático com explicações e sugestões de correção.

O objetivo geral do projeto é desenvolver uma ferramenta de linha de comando
capaz de revisar código Python com base em boas práticas de programação. Como
objetivos específicos, busca-se: receber código-fonte a partir de arquivos,
diretórios ou repositórios públicos do GitHub; validar automaticamente a
aderência às regras PEP 8; estruturar os resultados por arquivo, linha, coluna e
categoria; gerar feedback em linguagem natural; e registrar métricas que
permitam avaliar a consistência entre a saída determinística e o texto produzido
pela IA.

A justificativa do trabalho está no uso complementar de abordagens simbólicas e
generativas. A análise estática oferece auditabilidade e reduz ambiguidades,
enquanto modelos de linguagem podem melhorar a apresentação do feedback,
explicando causas e alternativas de correção. O projeto também se alinha à
opção LLM12 da disciplina, pois emprega grandes modelos de linguagem para
analisar dados técnicos de uma atividade prática: o código-fonte. Neste
contexto, o código é tratado como artefato de entrada, e o agente produz uma
análise estruturada sobre sua qualidade em relação a um conjunto explícito de
boas práticas.

O restante do artigo está organizado da seguinte forma. A Seção 2 apresenta a
fundamentação teórica e os trabalhos relacionados. A Seção 3 descreve a
metodologia, a estrutura dos dados, os critérios de avaliação e as próximas
etapas. A Seção 4 discute os resultados obtidos, as limitações observadas e os
aspectos éticos do uso de IA. Por fim, a Seção 5 apresenta as conclusões e
contribuições do trabalho.

\section{Fundamentação Teórica}

\subsection{Qualidade de código, PEP 8 e legibilidade}

Qualidade de código é um conceito multidimensional, associado a atributos como
legibilidade, manutenibilidade, testabilidade, reuso, simplicidade e aderência a
padrões de desenvolvimento. Em revisão moderna de código, revisores não avaliam
apenas a presença de defeitos funcionais; também observam clareza, estilo,
impacto de manutenção e adequação às convenções do projeto
\cite{bacchelli2013modernreview}. Em programação educacional, esses atributos
ganham relevância porque a forma como o estudante escreve o código interfere na
compreensão futura do próprio raciocínio e na comunicação com colegas e
professores.

A PEP 8 estabelece convenções de estilo para Python, incluindo espaçamento ao
redor de operadores, linhas em branco entre definições, limite recomendado de
comprimento de linha, organização de importações e convenções de nomenclatura
\cite{vanrossum2001pep8}. Essas regras não substituem projeto de software,
testes ou revisão humana, mas reduzem variação desnecessária na escrita do
código. Para estudantes, esse tipo de padronização cria um vocabulário comum de
boas práticas e facilita a identificação de padrões recorrentes de erro.

\subsection{Análise estática e ferramentas de lint}

Análise estática examina artefatos de software sem executar o programa. No
contexto deste projeto, a análise estática é utilizada para identificar
violações de estilo de forma automática, repetível e independente da
interpretação subjetiva de um revisor. A ferramenta pycodestyle
implementa verificações relacionadas à PEP 8 e retorna diagnósticos com código
da violação, linha, coluna e descrição textual \cite{pycqa2026pycodestyle}.
Exemplos de categorias verificadas são E225, para ausência de espaços ao redor
de operadores, E231, para ausência de espaço após vírgula, e E302, para falta
de linhas em branco antes de definições no nível do módulo.

A vantagem dessa abordagem é a alta rastreabilidade: cada apontamento pode ser
associado a uma posição específica do código. A limitação é que o feedback
produzido por linters tende a ser curto e técnico. Assim, embora a ferramenta
determine \textit{o que} está em desacordo com a regra, ela nem sempre explica
\textit{por que} aquilo prejudica a legibilidade ou \textit{como} um estudante
pode corrigir o problema sem apenas decorar códigos de erro.

\subsection{Modelos de linguagem e agentes orientados por ferramentas}

Modelos de linguagem de grande porte ampliaram as possibilidades de automação
em tarefas de engenharia de software, especialmente na geração de explicações,
resumos, sugestões e comentários de revisão. Entretanto, esses modelos podem
produzir respostas variáveis, inferências incorretas ou recomendações não
ancoradas nos dados de entrada. Estudos recentes sobre agentes baseados em LLMs
destacam a importância de integrar modelos generativos a ferramentas externas,
memória, planejamento e fontes verificáveis de informação
\cite{wang2024agents,plaat2025agentic}. Essa perspectiva favorece arquiteturas
híbridas, nas quais o modelo não atua sozinho, mas interpreta resultados
produzidos por componentes mais determinísticos.

O projeto adota essa lógica. O agente não solicita ao modelo que descubra
livremente todos os problemas do código. Em vez disso, primeiro executa
pycodestyle; depois, envia ao modelo o código-fonte e a lista
estruturada de violações. Com isso, a LLM é usada principalmente como camada de
explicação e comunicação. A decisão reduz o risco de contagens inventadas,
melhora a auditabilidade do processo e permite comparar a resposta textual com
um gabarito objetivo.

\subsection{Feedback automatizado em programação}

Ferramentas de avaliação automática são frequentes no ensino de programação,
sobretudo para verificar corretude por testes unitários ou comparar saídas
esperadas. Revisões sistemáticas indicam, porém, que muitos sistemas ainda
focam mais em correção funcional do que em atributos de legibilidade,
manutenibilidade e documentação \cite{messer2024grading}. Essa lacuna é
relevante porque estudantes podem entregar soluções corretas, mas difíceis de
ler e manter.

Pesquisas sobre feedback automatizado em tarefas introdutórias de programação
mostram que retornos rápidos podem apoiar ciclos de tentativa, erro e
aprendizagem, desde que o feedback seja específico e acionável
\cite{singh2013feedback}. Mais recentemente, trabalhos sobre feedback formativo
de qualidade de código reforçam a importância de apresentar problemas de forma
compreensível, priorizada e conectada a métricas ou regras explícitas
\cite{vandenaker2024feedback}. O presente projeto se insere nessa direção ao
usar diagnósticos PEP 8 como base objetiva para explicações produzidas por IA.

\section{Metodologia e Resultados Esperados}

\subsection{Escopo e arquitetura da solução}

A solução foi implementada em Python e executada por linha de comando. O sistema
aceita três modos de entrada mutuamente exclusivos: arquivos locais
(\texttt{-f}), diretórios locais (\texttt{-d}) e repositórios públicos do GitHub
(\texttt{-g}). No modo GitHub, o programa interpreta a URL informada, consulta a
API de conteúdos do GitHub, percorre arquivos e diretórios, filtra artefatos
\texttt{.py} e respeita limites configuráveis de quantidade e tamanho de arquivo
\cite{github2026contents}. Em todos os modos, a análise é restrita a arquivos
Python, pois o critério de qualidade adotado nesta etapa é a conformidade com a
PEP 8.

A arquitetura possui cinco componentes principais: leitura e filtragem dos
arquivos; validação PEP 8 com pycodestyle; estruturação dos diagnósticos;
montagem do prompt; e geração do relatório via Ollama. O Ollama
é utilizado como interface para comunicação com o modelo de linguagem por meio
de chamada de chat \cite{ollama2026docs}. O modelo não foi treinado ou ajustado
com dados próprios neste estágio; ele atua como gerador condicionado pelo
conteúdo do código e pela lista de violações calculada previamente.

\subsection{Pipeline de processamento}

O pipeline inicia com o parser de argumentos, que identifica a origem do código
e parâmetros opcionais, como modo verboso e limites para coleta em repositórios.
Em seguida, o sistema transforma as entradas em uma estrutura uniforme:
\texttt{\{nome\_arquivo: conteudo\}}. Essa normalização permite que arquivos
vindos de fontes diferentes sejam processados pela mesma sequência de etapas.

Na etapa de filtragem, apenas arquivos com extensão \texttt{.py} são mantidos.
No caso de diretórios, a busca é recursiva e ignora pastas como \texttt{.git},
\texttt{.venv}, \texttt{\_\_pycache\_\_}, \texttt{dist} e \texttt{build}. Essa
decisão reduz ruído, evita analisar dependências ou artefatos gerados e torna a
saída mais coerente com o objetivo pedagógico.

Após a filtragem, cada arquivo é submetido ao pycodestyle. Como a
biblioteca opera sobre caminhos de arquivo, o conteúdo é escrito
temporariamente em um arquivo \texttt{.py}, analisado e removido ao final da
verificação. Para cada violação, são coletados linha, coluna, código e mensagem.
As ocorrências são ordenadas por posição e categoria, favorecendo
reprodutibilidade e comparação posterior com o gabarito.

Com os diagnósticos consolidados, o programa monta um prompt contendo o código
integral, a lista de violações e um resumo por categoria. O prompt inclui
restrições explícitas: a LLM deve basear contagens exclusivamente nas violações
fornecidas, não inventar códigos de erro ou linhas inexistentes e marcar como
sugestão extra qualquer observação que extrapole o escopo do
pycodestyle. Por fim, a resposta é salva em um relatório
Markdown, permitindo leitura posterior e comparação com o resultado
determinístico.

\subsection{Dataset e análise exploratória}

O conjunto de avaliação contém 25 arquivos Python armazenados em
\texttt{tests/arquivospy}. Os arquivos simulam exercícios introdutórios, como
calculadora, manipulação de strings, listas, classes simples, pilha, fila,
geometria, datas, criptografia, senhas, matrizes e cadastro de alunos. O
dataset foi construído para conter arquivos pares com violações intencionais e
arquivos ímpares sem violações, gerando um cenário controlado para avaliar
detecção e explicação de erros.

A análise exploratória mostrou que 13 arquivos não apresentaram violações e 12
arquivos concentraram todos os problemas. Foram identificadas 217 violações PEP
8. As duas categorias mais frequentes foram E231, ausência de espaço após
vírgula, com 78 ocorrências, e E225, ausência de espaço ao redor de operadores,
com 62 ocorrências. Juntas, essas duas categorias representam aproximadamente
64,5\% das violações. Esse padrão é coerente com erros de formatação comuns em
programadores iniciantes, que tendem a escrever expressões e chamadas de função
corretas do ponto de vista sintático, mas ainda pouco padronizadas.

\subsection{Critérios de avaliação}

A avaliação do agente foi definida a partir de quatro critérios. O primeiro é a
consistência quantitativa, isto é, a correspondência entre as contagens do
pycodestyle, do gabarito e do relatório gerado. O segundo é a cobertura
de categorias, observando se todos os códigos de erro detectados aparecem no
feedback. O terceiro é a qualidade explicativa, analisando se a resposta
descreve a regra violada e apresenta uma correção compreensível. O quarto é a
confiabilidade textual, avaliando se o modelo evita afirmações que contradigam
os dados determinísticos de entrada.

Esses critérios separam duas responsabilidades do sistema. A detecção de
violações pertence à etapa determinística e pode ser medida objetivamente. A
explicação das violações pertence à etapa generativa e exige inspeção
qualitativa. Essa separação é importante porque uma resposta textual agradável
não deve ser confundida com uma análise tecnicamente correta.

\subsection{Resultados esperados e próximas etapas}

Espera-se que o agente preserve integralmente as violações detectadas pelo
pycodestyle e produza feedback claro para estudantes. Também se espera
que a estruturação do prompt reduza inconsistências comuns em respostas
gerativas, especialmente contagens divergentes e comentários sem base nos dados
de entrada.

As próximas etapas previstas são: implementar um avaliador automático que compare
o relatório da LLM com o gabarito; gerar o resumo global de forma determinística,
sem depender da aritmética do modelo; ampliar o dataset com códigos reais de
disciplinas introdutórias, preservando anonimização quando necessário; comparar
a solução com ferramentas como ruff, flake8, black e
SonarQube; coletar avaliação de utilidade do feedback com estudantes ou
monitores; e disponibilizar uma interface web simples para uso em sala de aula
ou monitoria.

\section{Resultados, Ética e Discussão}

\subsection{Resultados obtidos}

A execução do pycodestyle sobre os 25 arquivos do dataset identificou
217 violações PEP 8 em 12 arquivos. Os 13 arquivos restantes não apresentaram
violações. Considerando todos os arquivos, a média foi de 8,68 violações por
arquivo; considerando apenas os arquivos com problemas, a média foi de 18,08
violações por arquivo. O arquivo com maior número de violações foi
\texttt{24\_logger.py}, com 41 ocorrências.

\subsection{Exemplo de saída e análise qualitativa}

Para um erro E225, por exemplo, o relatório gerado explica que operadores devem
ser separados por espaços e apresenta uma comparação entre formas incorreta e
correta, como \texttt{cont=0} e \texttt{cont = 0}. Para erros E201 e E202, o
relatório explica a remoção de espaços internos em chamadas ou definições, como
\texttt{func( valor )} para \texttt{func(valor)}. Esse tipo de explicação
aproxima a saída técnica do linter de uma orientação formativa.

Também foram observadas sugestões extras, como uso de \textit{type hints},
\texttt{@dataclass}, testes unitários e bibliotecas padrão. Essas sugestões
podem enriquecer o feedback, mas precisam ser claramente separadas das violações
PEP 8. Caso contrário, o estudante pode interpretar uma melhoria opcional como
erro objetivo. Por isso, o prompt foi ajustado para exigir marcação explícita de
recomendações fora do escopo do pycodestyle.

\subsection{Discussão crítica}

O principal avanço observado é a separação entre detecção e explicação. A
detecção feita por pycodestyle foi reprodutível e gerou métricas
objetivas. A LLM agregou valor ao traduzir códigos de erro em texto didático,
mas apresentou risco de inconsistência em sínteses globais. Em uma saída
intermediária, por exemplo, o resumo textual informou quantidade incorreta de
arquivos limpos e arquivos com violações, embora as contagens por arquivo
estivessem corretas. Esse caso confirma a necessidade de manter métricas globais
fora da geração livre do modelo.

Outra limitação é que a PEP 8 cobre apenas parte das boas práticas de
programação. O agente atual não avalia corretude funcional, segurança,
complexidade algorítmica, cobertura de testes, duplicação de lógica ou qualidade
arquitetural. Assim, os resultados devem ser interpretados como uma revisão de
estilo e legibilidade inicial, não como certificação completa de qualidade de
software.

\subsection{Aspectos éticos e responsabilidade no uso da IA}

O uso de IA em revisão de código exige cuidado com privacidade, autoria,
dependência excessiva e transparência. Como código-fonte pode conter lógica de
negócio, dados sensíveis ou propriedade intelectual, o projeto foi desenhado
para funcionar com uma camada de inferência configurável via Ollama. Ao
utilizar modelos locais, o código analisado permanece no ambiente do usuário.
Quando modelos remotos ou em nuvem forem configurados, essa condição deve ser
explicitada aos usuários antes da análise.

Também é necessário evitar que o feedback automatizado seja tratado como
avaliação final e incontestável. A ferramenta deve apoiar estudantes e
professores, não substituir revisão humana. Em contexto educacional, recomenda-se
que o agente seja usado como pré-revisão formativa, ajudando o estudante a
corrigir problemas recorrentes antes de uma avaliação mais ampla sobre lógica,
estrutura e compreensão do problema.

\section{Conclusão}

Este trabalho apresentou um agente de IA para revisão de código Python baseado
em boas práticas de programação. A solução combina validação determinística com
pycodestyle e geração de feedback em linguagem natural por meio de um
modelo acessado via Ollama. A arquitetura proposta permite analisar
arquivos locais, diretórios e repositórios públicos do GitHub, mantendo uma
separação clara entre a etapa objetiva de detecção e a etapa generativa de
explicação.

Os resultados obtidos no dataset de 25 arquivos indicam que o objetivo principal
foi alcançado. O pipeline identificou 217 violações PEP 8 em 12 arquivos e
produziu dados suficientes para análise quantitativa e qualitativa. A principal
contribuição do projeto é mostrar que uma abordagem híbrida pode tornar a saída
de ferramentas de lint mais acessível para estudantes, preservando
rastreabilidade técnica. Como evolução, o trabalho deve incorporar avaliação
automática do relatório gerado, ampliar a base de testes, comparar ferramentas
de análise estática e validar a utilidade do feedback com usuários reais.
